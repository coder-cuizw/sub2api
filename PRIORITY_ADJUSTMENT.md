# 优先级调整：基于统一代理出口IP

## 原始设计的变化

**你的架构**：
```
下游客户端（多个IP）
        ↓
    你的网关
        ↓
    你的代理IP（单一固定IP）
        ↓
  Anthropic API
```

**Anthropic 的视角**：
- 所有请求都来自同一个 IP
- 同一 IP 下的多个 token → 多个账号从同一个代理 IP 出去（正常）
- 同一 token 的多个虚拟 device_id → 同一用户的多个设备（正常）

---

## 风险重新评估

### 被降低风险的项

#### ❌ 不用再管：下游 IP 多源问题
```
原本的风险：
  Anthropic 看到同一 token 从 IP-A、IP-B、IP-C 发来
  → "这个账号从多个地方被访问"
  
现在的情况：
  所有请求都从 proxy-ip 出去
  → Anthropic 看到"这个账号一直在用代理，很正常"
  → 风险消除 ✅
```

**可以删除的防护**：
- 下游 IP 频率限制（已不需要，因为 Anthropic 看不到下游 IP）
- 下游 IP 异常行为检测（同理）
- "下游滥用隔离"的概念（Anthropic 无法区分下游来源）

---

#### ❌ 风险大幅降低：代理 IP 信誉问题

**原本**：多个下游 IP 发请求可能触发 Anthropic 的风险控制

**现在**：Anthropic 只看到你的代理 IP
- 如果代理 IP 是 residential IP（住宅代理）：✅ 风险低
- 如果代理 IP 是 datacenter IP（数据中心代理）：⚠️ 风险中等
- 如果频繁切换代理 IP：⚠️ 风险高

**建议**：选择**稳定的单一代理 IP**，最好是 residential IP

---

### 新增/保留的关键风险

#### 🔴 P0：指纹续期并发问题（风险**上升**）

**为什么风险上升**？

在原本的"下游多IP"架构中，不同下游IP的并发请求可能：
- 分布在不同时刻，不太会同时触发指纹续期

在你的"单一代理IP"架构中：
- 所有并发请求都从同一代理IP出去
- 多个并发请求可能**同时**触发指纹续期
- 导致 device_id 在极短时间内振荡（毫秒级）
- Anthropic 更容易检测到

```
时刻 T:
  Request 1: 触发指纹续期，生成 device_id = "abc123"
  Request 2: 同时触发续期，生成 device_id = "def456"
  Request 3: 同时触发续期，生成 device_id = "ghi789"
  ↓
  Anthropic 在同一毫秒看到三个不同的 device_id
  → 明显的异常行为
```

**优先级**：上升到 **P0**（关键）

---

#### 🔴 P0：虚拟身份绑定周期的信息泄露（风险**上升**）

**为什么风险上升**？

如果不同的虚拟 device_id 使用同一代理 IP：
```
虚拟身份 A（device_id = "abc123"）：使用 device_ip = proxy_ip，创建于 Day 1
虚拟身份 B（device_id = "def456"）：使用 device_ip = proxy_ip，创建于 Day 8
虚拟身份 C（device_id = "ghi789"）：使用 device_ip = proxy_ip，创建于 Day 15

Anthropic 的关联分析：
  ABC 三个设备都在 proxy_ip 上
  且从未出现过其他 IP
  → "这三个'设备'其实来自同一个代理"
  → 可能是伪装
```

**优先级**：保持 **P1**（但需要特别注意）

## 新的防封重点

### 关键防护（按优先级）

**1. 指纹续期并发安全** 🔴 最关键
```
症状：多个并发请求同时触发续期
结果：device_id 在毫秒内变化多次
后果：Anthropic 识别为异常 → 标记账号
```

**修复**：
```go
// 使用分布式锁确保同时只有一个续期操作
if !cache.SetNXWithExpiry(fmt.Sprintf("fp_renew_%d", accountID), "renewing", 10*time.Second) {
    // 其他线程已在续期，等待其完成
    time.Sleep(100*time.Millisecond)
    return cache.GetFingerprint(accountID)  // 读取续期后的指纹
}

fp := renewFingerprint(accountID)
cache.SetFingerprint(accountID, fp)
cache.Del(fmt.Sprintf("fp_renew_%d", accountID))
```

---

**2. 使用模式随机化** 🟠 重要
```
症状：账号 24/7 不停运行，总是用满 200K context
后果：与真人使用模式差距太大 → 标记为机器
```

**修复**：
```go
// 模拟用户的活跃时间
func getActiveTimeModulation() float64 {
    hour := time.Now().UTC().Hour()
    
    // 假设用户在美国东部时区，工作时间 9-18，晚上降速，凌晨几乎停止
    switch {
    case hour >= 2 && hour <= 6:     // 2-6 AM：用户睡眠
        return 0.1  // 降速到 10%
    case hour >= 6 && hour <= 9:     // 6-9 AM：起床阶段
        return 0.5
    case hour >= 9 && hour <= 18:    // 9-18：工作时间
        return 1.0
    case hour >= 18 && hour <= 22:   // 18-22：下班
        return 0.7
    default:                         // 22-2：睡前
        return 0.3
    }
}

// 在超时 / 并发控制处使用
actualConcurrentLimit := maxConcurrent * int(getActiveTimeModulation())
```

**3. 账号冷却期** 🟠 重要
```
症状：账号被限流，继续猛打，加速封禁
后果：完全禁用 → 虚拟身份全部失效
```

**修复**：
```go
if upstreamResponse.StatusCode == 429 {  // Rate Limited
    account.EnterCooldown(1 * time.Hour)
    
    // 后续请求自动转向其他账号或返回友好错误
    // 不要再打这个账号
}
```

## 实际效果

有了这些简化的重点防护，你的防封能力从：

```
当前：覆盖 60% 的防封链路（有很多"可能有用"的防护）
      ↓
改进：覆盖 80% 的防封链路（专注于"真正有用"的防护）
      ↓
理想：覆盖 95% 的防封链路（加上用户行为模拟）
```

关键是：**不要过度设计，专注于最可能触发检测的三个点**：
1. 并发指纹振荡
2. 机器化行为模式
3. 持续滥用后的冷却
